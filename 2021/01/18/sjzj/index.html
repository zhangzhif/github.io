数据质检实战解析

2021-01-18
  • java

前言

解决多种数据源的数据质量检测

主要结构

Reader 模块

不同的数据源有不同的连接方式，reader模块主要是对不同的数据源提供接口规范，读取数据后封装成统一的数据格式，便于数据读取之后的统一处理

Handler 模块

提供统一的数据接收方式，自己实现处理逻辑

channal 模块

进行数据运输，可进行自定义限流方式（目前主要是字节大小）、监控范围、数据格式等

reader 就是进行数据生产、handler 进行数据消费。就像你在网上买了张电话卡（相当于数据），你只需要选择运营商、套餐等信息（数据格式）,通过快递或者人工送到你手上（传输方式）,
这个过程你不需要去考虑卡是哪个门店给你寄出的，而最后你把卡插到苹果手机还是华为手机上或者放放着不激活这也不是给你寄卡的人所需要知道的(hander）,下单到拿卡整个过程最重要的就是卡的规格和物流的速度

规范设计好我们的核心就是数据的传输方式了、这个地方也是数据质检性能的第一个提升点

问题

为什么reader不直接把数据放入channel中然后handler直接重里面拿

ArrayBlockingQueue是一个阻塞式的队列,那在插入和读取都很多时，就会造成时间浪费，所以先把数据放到自己的数据集合中，然后进行批量插入到队列当中，减少队列的操作次数以达到性能的提升

限流方式

根据数据条数限流

在数据进入channel之前进行判断（管道中已有的数据量+本次新增数据量>最大允许内存）如果满足reader线程进行阻塞，等待handler处理完一批数据后唤醒

根据数据所占内存大小限流

在数据进入channel之前进行判断（管道中已有数据内存+本次新增数据所需内存>最大允许内存）如果满足reader线程进行阻塞，等待handler处理完一批数据后唤醒

日志相关

之前每次处理数据量变动都会有一条日志信息产生使得日志文件过大占用过多的系统内存，本次新增ocean-message模块进行异步信息推送，页面数据量实时变动，提升用户体验
